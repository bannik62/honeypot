// de la structure à l'architecture

① Anatomie d'une requête HTTP

Une requête HTTP c'est un message texte structuré en 3 parties envoyé du client vers le serveur.

1
Request Line
Méthode + URL + Version HTTP
2
Headers
Métadonnées clé:valeur (auth, type de contenu, cookies…)
3
Body (optionnel)
Données envoyées (JSON, form-data, binaire…)
// Requête HTTP brute
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGci...
Content-Length: 42
                                       ← ligne vide obligatoire
{"name": "Yohann", "role": "dev"}

// Décomposer une URL

https://api.example.com:443/users/42?sort=asc&limit=10#section

https://          ← protocole (scheme)
api.example.com   ← hostname
:443              ← port (optionnel si standard)
/users/42         ← path (ressource)
?sort=asc&limit=10 ← query string (paramètres)
#section          ← fragment (côté client uniquement)
💡 Le fragment #anchor n'est jamais envoyé au serveur — il reste dans le browser.

② Méthodes HTTP

La méthode définit l'intention de la requête. Chaque méthode a une sémantique précise.

GET Lire une ressource. Sans effet de bord. Idempotent.
Le body est ignoré — toujours passer les params en query string.
POST Créer une ressource ou soumettre des données. Non-idempotent.
Appel multiple = effets multiples (double soumission = double création).
PUT Remplacer entièrement une ressource existante. Idempotent.
Envoyer l'objet complet. Ce qui manque sera écrasé/supprimé.
PATCH Modifier partiellement une ressource. Idempotent si bien implémenté.
Envoyer uniquement les champs à modifier.
DELETE Supprimer une ressource. Idempotent.
Un second DELETE sur une ressource déjà supprimée → 404 (pas d'erreur grave).

// Idempotence — concept clé

IDEMPOTENT ✓
Appeler N fois = même résultat que 1 fois.
GET, PUT, DELETE, PATCH
NON-IDEMPOTENT ✗
Chaque appel crée un effet nouveau.
POST

// Exemples REST

GET    /api/users        → liste tous les users
GET    /api/users/42     → récupère user id=42
POST   /api/users        → crée un user
PUT    /api/users/42     → remplace user id=42
PATCH  /api/users/42     → modifie partiellement user id=42
DELETE /api/users/42     → supprime user id=42

③ Headers

Les headers sont des métadonnées de la requête. Ils ne font pas partie du body mais conditionnent tout son traitement.

// Headers de requête essentiels

HEADERRÔLEEXEMPLE
Content-Type Format du body envoyé application/json
Accept Format attendu en réponse application/json
Authorization Auth token / credentials Bearer <token>
Cookie Cookies session session=abc123
User-Agent Identification du client Mozilla/5.0...
Origin Origine pour CORS https://monapp.fr

// Headers de réponse essentiels

HEADERRÔLE
Content-TypeFormat du body retourné
Set-CookieSetter un cookie côté client
Cache-ControlPolitique de cache
Access-Control-Allow-OriginCORS — origines autorisées
LocationRedirect URL (3xx)
X-RateLimit-RemainingRate limit restant
⚠️ Oublier Content-Type: application/json sur un POST → le serveur lira le body comme texte brut et le parsing JSON échouera silencieusement.

// Custom headers

// Convention : préfixe X- (deprecated mais encore courant)
X-Request-ID: 550e8400-e29b-41d4-a716

④ Body — formats

Le body porte les données. Son format est dicté par le header Content-Type.

// JSON (le plus courant)

Content-Type: application/json

{
  "user": "Yohann",
  "role": "dev",
  "tags": ["node", "svelte", "docker"]
}

// Form URL-encoded (formulaires HTML)

Content-Type: application/x-www-form-urlencoded

user=Yohann&role=dev&tags=node
// Les valeurs sont URL-encodées : espace → %20, & → %26

// Multipart / file upload

Content-Type: multipart/form-data; boundary=----XYZ

------XYZ
Content-Disposition: form-data; name="file"; filename="photo.jpg"
Content-Type: image/jpeg

[binary data]
------XYZ--

// Raw / binaire / autres

text/plain
Texte brut sans structure
application/octet-stream
Binaire brut (téléchargements)
application/xml
XML (SOAP, legacy APIs)
application/graphql
Requêtes GraphQL
💡 GET et DELETE n'ont techniquement pas de body. Certains serveurs l'ignorent. Ne jamais en envoyer pour ces méthodes.

⑤ Structure de la réponse

La réponse HTTP suit la même structure : status line + headers + body.

HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/users/42
X-Request-ID: uuid-xxxx

{"id": 42, "name": "Yohann", "created": true}

// Codes de statut

2xxSuccès
200OK — requête réussie
201Created — ressource créée (POST)
204No Content — succès sans body (DELETE)
3xxRedirection
301Moved Permanently — nouvelle URL définitive
302Found — redirect temporaire
304Not Modified — cache valide, pas de body
4xxErreur client
400Bad Request — body/params invalides
401Unauthorized — pas/mauvais token
403Forbidden — token valide, accès refusé
404Not Found — ressource inexistante
422Unprocessable — validation échouée
429Too Many Requests — rate limit atteint
5xxErreur serveur
500Internal Server Error — crash serveur
502Bad Gateway — proxy/reverse proxy cassé
503Service Unavailable — surcharge/maintenance
⚠️ 401 vs 403 : 401 = "qui es-tu ?" (pas/mauvais token). 403 = "je te connais mais non." (token valide, droits insuffisants).

⑥ HTTPS — TLS en pratique

HTTPS = HTTP + TLS. La couche TLS chiffre tout le trafic entre client et serveur.

// TLS Handshake simplifié

1
Client Hello
Le client envoie les algos de chiffrement supportés + un nonce aléatoire
2
Server Hello + Certificat
Le serveur choisit l'algo + envoie son certificat TLS (signé par une CA)
3
Vérification cert + clé de session
Le client vérifie la chaîne de confiance et génère une clé de session symétrique
4
Échange chiffré
Tout le reste est chiffré avec AES (symétrique, rapide)

// Ce que TLS protège / ne protège pas

PROTÉGÉ ✓
Headers, body, query string, cookies, tous chiffrés en transit
PAS PROTÉGÉ ✗
Le hostname (SNI visible en clair). L'IP de destination. Les métadonnées réseau.

// Certificats en pratique (VPS)

# Let's Encrypt via Certbot (Apache)
certbot --apache -d mondomaine.fr -d www.mondomaine.fr

# Renouvellement auto (cron)
certbot renew --quiet

# Vérifier l'expiration
openssl s_client -connect mondomaine.fr:443 -servername mondomaine.fr \
  2>/dev/null | openssl x509 -noout -dates

// SNI — Virtual Hosting HTTPS

SNI (Server Name Indication) permet à un serveur d'héberger plusieurs domaines HTTPS sur la même IP. Le client envoie le hostname en clair pendant le handshake TLS pour que le serveur sache quel certificat présenter.

💡 Le LAB force le SNI en Python avec server_hostname=target_host — c'est exactement ça.

⑦ Organiser ses requêtes

Côté code, l'organisation des appels HTTP conditionne la maintenabilité. Voici les patterns.

// Pattern 1 — Couche API centralisée

// lib/api.js — instance axios configurée une seule fois
import axios from 'axios'

const api = axios.create({
  baseURL: import.meta.env.VITE_API_URL,
  timeout: 10000,
  withCredentials: true,
})

api.interceptors.request.use(config => {
  const token = getCSRFToken()
  if (token) config.headers['X-CSRF-Token'] = token
  return config
})

api.interceptors.response.use(
  res => res,
  err => {
    if (err.response?.status === 401) logout()
    return Promise.reject(err)
  }
)

export default api

// Pattern 2 — Modules par ressource

// services/users.js
export const usersService = {
  getAll:  (params)   => api.get('/users', { params }),
  getById: (id)       => api.get(`/users/${id}`),
  create:  (data)     => api.post('/users', data),
  update:  (id, data) => api.patch(`/users/${id}`, data),
  remove:  (id)       => api.delete(`/users/${id}`),
}

// Pattern 3 — Retry + timeout

async function fetchWithRetry(url, opts = {}, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      const res = await fetch(url, { ...opts, signal: AbortSignal.timeout(5000) })
      if (!res.ok) throw new Error(`HTTP ${res.status}`)
      return await res.json()
    } catch (e) {
      if (i === retries - 1) throw e
      await sleep(500 * (i + 1))
    }
  }
}

// Bonnes pratiques

1. Centraliser la config — baseURL, headers par défaut, timeout dans un seul endroit.
2. Intercepteurs globaux — auth token, refresh token, gestion 401/403 au même endroit.
3. Modules par ressource — un fichier users.service.js par entité, pas de fetch éparpillé.
4. Toujours un timeout — sans timeout, une requête bloquée pend indéfiniment.
5. Distinguer 4xx et 5xx — 4xx = bug côté client à corriger. 5xx = alerter + retry possible.
